# 十二、SRE 方法论

## 9.1 SLO / SLI / Error Budget

| 概念 | 说明 | 示例 |
|:----|:------|:------|
| **SLI** | 服务级别指标，实际测量值 | 请求延迟 P99 = 200ms |
| **SLO** | 服务级别目标，约定的可靠性 | P99 < 300ms，可用性 99.9% |
| **Error Budget** | 允许的不可靠额度 | 每月允许 43 分钟的故障时间 |

### 计算

```
SLO = 99.9% 可用性
Error Budget = (100% - 99.9%) × 30天 × 24小时 × 60分
             = 0.1% × 43200分钟
             = 43.2 分钟/月
```

### Error Budget 的价值

- Error Budget 耗尽 → 暂停新功能发布，全力修复稳定性
- Error Budget 充足 → 可以加快发布节奏，承担更多变更风险
- 不是"不许出错"，而是"在一定范围内允许出错"

## 9.2 故障演练

### 混沌工程

主动注入故障，验证系统的韧性。

| 工具 | 特点 |
|:----|:------|
| **Chaos Mesh** | K8s 原生，Pod Kill / 网络延迟 / IO 注入 |
| **Litmus** | K8s 混沌工程平台 |
| **Gremlin** | 商业产品，跨平台 |

### 演练场景

| 场景 | 注入方式 | 验证目标 |
|:----|:------|:------|
| Pod 被 Kill | `kubectl delete pod` | 自动恢复、请求无中断 |
| 网络延迟 | chaos-mesh 注入 500ms 延迟 | 超时重试机制生效 |
| 节点宕机 | 停止某个节点 | 多副本跨节点、故障转移 |
| 数据库主从切换 | 手动触发切换 | 连接池重连、业务无感知 |
| 全链路压测 | JMeter / Locust | 容量规划、限流熔断 |

## 9.3 On-Call 机制

### 事件分级

| 级别 | 定义 | 响应时间 | 通知方式 |
|:----|:------|:----:|:------|
| **P0** | 服务全面不可用 | 5 分钟 | 电话 + 群消息 |
| **P1** | 核心功能受损 | 15 分钟 | 群消息 + 电话 |
| **P2** | 非核心异常 | 1 小时 | 群消息 |
| **P3** | 潜在风险 | 工作时间 | 消息 |

### 值班轮转

- 明确每个时段的责任人
- 主 On-Call + 副 On-Call（备份）
- 跨时区团队衔接，避免熬夜

### 事后复盘

**Blameless Postmortem 结构：**

1. **事件概述**：什么时间、什么影响、持续多久
2. **时间线**：从发现到恢复的关键时间节点
3. **根因分析**：5 Whys 方法
4. **影响**：用户影响范围、Error Budget 消耗
5. **改进措施**：短期修复 + 长期预防，落实到具体人和时间

## 9.4 容量规划

| 步骤 | 说明 |
|:----|:------|
| **负载建模** | 历史流量分析 + 业务预期增长 |
| **资源分析** | 单实例 QPS、CPU/内存消费 |
| **弹性伸缩** | HPA/VPA 自动伸缩 + 集群弹性节点 |
| **压测验证** | 以 1.5~2x 峰值流量压测，确认系统承载上限 |
| **成本优化** | Spot 实例、资源缩容、闲置检查 |

## 9.5 可靠性度量

| 指标 | 计算方式 |
|:----|:------|
| **可用性** | (总时间 - 故障时间) / 总时间 × 100% |
| **MTBF** | 平均无故障时间 |
| **MTTR** | 平均恢复时间 |
| **变更失败率** | 导致回滚或故障的变更 / 总变更数 |
| **部署频率** | 每天/每周的部署次数 |

> 四个关键指标（DORA Metrics）：部署频率、变更前置时间、变更失败率、恢复时间。
