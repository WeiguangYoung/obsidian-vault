# 🚗 UDS（统一诊断服务）入门指南

## 什么是 UDS？

**UDS（Unified Diagnostic Services，统一诊断服务）** 是国际标准 **ISO 14229** 定义的汽车电子控制单元（ECU）诊断协议。说白了，就是**修车电脑跟汽车各个控制器「对话」的标准化语言**。

> **一句话：** UDS 是汽车界的 TCP/IP — 不管哪个厂家、哪个 ECU，都用同一套「语言」来做诊断。

---

## 为什么需要 UDS？

现代汽车里有 **几十甚至上百个 ECU**（发动机、变速箱、ABS、安全气囊、空调、门窗、电池管理……）。每个 ECU 都可能需要：

- 🔍 **读取故障码** — 哪里坏了？
- 📊 **读取实时数据** — 现在转速多少？温度多少？
- ✏️ **写入配置** — 刷写新固件、配置功能参数
- 🔄 **执行特殊操作** — 像"学习"节气门位置、激活某个执行器

如果没有统一标准，每个厂家各搞一套协议，那维修和开发就乱套了。**UDS 就是汽车行业统一的诊断语言。**

---

## UDS 的核心概念

### 1️⃣ 请求-响应模型

UDS 是典型的 **Client-Server 架构**：

```
诊断仪 (Client)  ──── 请求 ────►  ECU (Server)
诊断仪 (Client)  ◄──── 响应 ────  ECU (Server)
```

- **Client**：诊断仪（OBD 扫描工具、刷写工具、产线设备）
- **Server**：ECU（发动机控制器、电池管理器 BMS……）

### 2️⃣ 服务 (Service) / SID

UDS 定义了 **26 种标准服务**，每种服务用一个 **SID（Service ID）** 标识（1 字节）。最常用的：

| SID | 名称 | 作用 |
|:---:|:----|:-----|
| **0x10** | Diagnostic Session Control | 切换诊断会话（默认/编程/扩展） |
| **0x11** | ECU Reset | 复位 ECU |
| **0x14** | Clear Diagnostic Information | 清除故障码 |
| **0x19** | Read DTC Information | 读取故障码 |
| **0x22** | Read Data By Identifier | 按 DID 读数据（版本号、电压等） |
| **0x2E** | Write Data By Identifier | 按 DID 写数据（配置参数） |
| **0x27** | Security Access | 安全访问（解锁 ECU 写入权限） |
| **0x28** | Communication Control | 通信控制（停用/启用报文） |
| **0x31** | Routine Control | 执行例程（自检、标定学习等） |
| **0x34** | Request Download | 请求下载（刷写前协商） |
| **0x36** | Transfer Data | 传输数据（刷写数据块） |
| **0x37** | Request Transfer Exit | 传输结束 |
| **0x3E** | Tester Present | 保持诊断会话活着 |

### 3️⃣ DID（Data Identifier）

DID 是一个 **2 字节的 ID**，用来标识具体的诊断数据项。例如：

| DID | 含义 |
|:---:|:-----|
| 0xF190 | 系统供应商 ID |
| 0xF191 | ECU 硬件版本 |
| 0xF192 | 软件版本号 / 刷写号 |
| 0x0100 | 车速（km/h） |
| 0x0101 | 发动机转速 |

> 用服务 `0x22` + DID `0xF192` 读软件版本：
> 请求：`22 F1 92`
> 响应：`62 F1 92 01 02 03`（版本号 V1.2.3）

### 4️⃣ DTC（Diagnostic Trouble Code）

故障码，告诉修理工「哪里坏了」。**3 字节** 组成：

| 字节 | 含义 | 示例 |
|:---:|:-----|:-----|
| 字节1（高） | 系统类型 | `0xP0`=动力总成, `0xB0`=车身 |
| 字节1（低）+ 字节2 | 故障部位 | `0x0101`=氧传感器 |
| 字节3 | 故障类型 | `0x04`=信号异常, `0x07`=机械故障 |

比如 **P0101** → `0x01 0x01` → 动力总成 + 氧传感器 + 信号异常。

### 5️⃣ 诊断会话（Session）

ECU 提供**多级会话模式**：

| 会话 | 服务 0x10 子功能 | 说明 |
|:----|:--------------:|:-----|
| **默认会话** | 0x01 | 上电默认状态，只能读不能写 |
| **编程会话** | 0x02 | 刷写固件用 |
| **扩展会话** | 0x03 | 写入参数、执行特殊操作 |
| **安全系统诊断** | 0x04 | 安全相关（气囊等） |

> 比如要刷写固件：先切到编程会话（0x10 02）→ 安全解锁（0x27）→ 请求下载（0x34）→ 传数据（0x36）→ 结束（0x37）

## UDS 报文结构（基于 CAN 总线）

最常见的是走 **CAN（Controller Area Network）总线**。一个典型的 UDS on CAN 报文：

```
┌──────────────────────────────────────────────────┐
│ CAN ID  │ PCI (协议控制信息)  │  SID  │  数据    │
├──────────────────────────────────────────────────┤
│ 0x7E0   │ 0x02               │ 0x22  │ F1 92   │
└──────────────────────────────────────────────────┘
```

- **CAN ID**：`0x7E0`（功能寻址）或 `0x7E0-0x7E7`（物理寻址），行业惯例
- **PCI**：报文长度（这个例子 = 2 字节有效数据）
- **SID + 数据**：实际诊断命令

### 三种寻址方式

| 方式 | 说明 | CAN ID 示例 |
|:----|:-----|:----------:|
| **物理寻址** | 一对一，指定某个 ECU | 请求 `0x7E0` → 响应 `0x7E8` |
| **功能寻址** | 一对多，广播给所有 ECU | 请求 `0x7DF`（所有 ECU 同时响应）|
| **扩展寻址** | 超过 8 字节时的分段传输 | 需要多帧传输 |

## UDS 刷写流程（典型场景）

刷写固件是 UDS 最常用的场景，完整流程：

```
步骤1: 切换编程会话     ── 0x10 02
步骤2: 安全解锁         ── 0x27 03 + 种子 → 0x27 04 + 密钥
步骤3: 检查编程前提     ── 0x31 01（例程控制：检查条件）
步骤4: 请求下载         ── 0x34（告知数据大小/地址）
步骤5: 传输数据块       ── 0x36（数据帧，可能几百块）
步骤6: 请求传输结束     ── 0x37
步骤7: 检查完整性       ── 0x31（校验和/CRC 验证）
步骤8: 执行复位         ── 0x11 01（复位 ECU，新固件生效）
```

## 小实战：用 Python 发送 UDS 请求

```python
import can

# 初始化 CAN 总线
bus = can.interface.Bus(bustype='socketcan', channel='can0')

# 发送 UDS 请求: 读取 DID 0xF192（软件版本）
# 服务 0x22 + DID F1 92
request = can.Message(
    arbitration_id=0x7E0,   # 物理请求 ID
    data=[0x22, 0xF1, 0x92],
    is_extended_id=False
)
bus.send(request)

# 接收响应（CAN ID 0x7E8）
response = bus.recv(timeout=1.0)
print(f"响应: {response.data.hex()}")
# 输出: 62 F1 92 01 02 03  → 软件版本 V1.2.3
```

> ⚠️ 实车测试需要连接 CAN 接口（如 PCAN-USB、Kvaser、SocketCAN）

## AUTOSAR 与 UDS 的关系

### 什么是 AUTOSAR？

**AUTOSAR（AUTomotive Open System ARchitecture）** 是全球汽车行业联合制定的**标准化软件架构**。可以理解成**汽车界的 Android** — 无论哪家芯片、哪个厂家，软件模块都能按统一规范对接。

> **一句话：AUTOSAR 定义了车控软件「怎么写」，UDS 定义了「怎么修」**

### AUTOSAR 分层架构

```
┌────────────────────────────────────────────────────┐
│              应用层 (Application Layer)              │
│   (车窗控制、雨刮逻辑、电池管理……业务代码在这里)          │
├────────────────────────────────────────────────────┤
│              运行时环境 (RTE)                         │
│   (虚拟功能总线，应用层与底层解耦的桥梁)                  │
├─────────┬──────────┬──────────┬────────────────────┤
│ 服务层   │  ECU     │  复杂    │                     │
│(Services)│  抽象层   │  驱动    │    微控制器抽象层     │
│          │(ECU Abst.)│(Complex) │    (MCAL)           │
│          │          │  Drivers │                     │
├─────────┴──────────┴──────────┴────────────────────┤
│               微控制器 (MCU)                         │
│    (Infineon TC3xx, NXP S32K, Renesas RH850 ...)    │
└────────────────────────────────────────────────────┘
```

### AUTOSAR 中的【CP】vs【AP】

汽车行业有**两套** AUTOSAR 标准，方向完全不同：

| 对比 | AUTOSAR CP (Classic) | AUTOSAR AP (Adaptive) |
|:----|:--------------------|:---------------------|
| **场景** | 硬实时控制 | 高性能计算 |
| **ECU 例子** | 发动机、ABS、气囊、门窗 | 自动驾驶域控、座舱域控 |
| **操作系统** | OSEK/FreeRTOS（无 MMU） | POSIX (Linux/QNX) |
| **编程语言** | C | C++ |
| **通信** | CAN/LIN/FlexRay | SOME/IP (以太网) |
| **诊断** | UDS on CAN | UDS on DoIP（以太网） |
| **OTA** | 弱 | 强（安全刷写原生支持） |
| **代表芯片** | Infineon TC3xx, NXP S32K | NXP S32G, TI TDA4, 高通 SA8295 |

> **简单记：CP = 硬实时控制，AP = 高性能计算**

### DCM — AUTOSAR 里谁管 UDS？

在 AUTOSAR CP 架构中，**DCM（Diagnostic Communication Manager）** 是专门负责 **UDS 协议栈**的模块：

```
┌─────────────────────────────────────┐
│  应用层                              │
│  ┌─ SWC 1 ─┐  ┌─ SWC 2 ─┐           │
│  │ 读DTC   │  │ 写DID   │           │
│  └─────────┘  └─────────┘           │
├─────────────────────────────────────┤
│          RTE                         │
├────────────────┬────────────────────┤
│  DCM           │  DEM (诊断事件管理) │
│ (UDS 协议解析) │ (DTC 存储/管理)     │
├────────────────┴────────────────────┤
│  CAN 栈 (CAN 驱动 / PduR / CanTp)   │
└─────────────────────────────────────┘
```

**DCM 的职责：**
1. 接收诊断请求报文（如 `22 F1 92` 读软件版本）
2. 解析 SID → 判断该服务是否在当前会话支持
3. 调用下层功能（读内存、写配置、控制 I/O）
4. 组装诊断响应报文发回

**诊断链路全流程：**
```
诊断仪 ──CAN──► CanTp (传输层，处理多帧分包)
                    └─► PduR (路由层，分发报文)
                           └─► DCM (解析 UDS 请求)
                                  ├─► 调用 SWC 接口读数据
                                  └─► 返回响应 → 反向回诊断仪
```

### 为什么搞 CICD 需要了解 AUTOSAR？

奇瑞的 CICD 岗位（以及大多数汽车嵌入式 DevOps 岗位），CI/CD 流水线不光是编译代码，还要：

- **刷写**：编译产物通常是 AUTOSAR 生成的 `.hex` / `.s19` / `.elf` 文件，需要通过 UDS 刷写到 ECU
- **配置**：AUTOSAR 的配置（SWC 接口、DID 定义、DTC 定义）存在 `arxml` 文件里，构建流程需要处理这些配置代码生成
- **测试**：HIL 测试中，诊断功能是必测项（UDS 服务的正确性验证）
- **版本管理**：每个 ECU 的软件版本号往往是通过 UDS DID 读取的，CI/CD 需要管理这些版本号的一致性

## Bootloader 与 UDS 的关系

### 什么是 Bootloader？

**Bootloader（引导加载程序）** 是 ECU 上电后最先运行的一段**不可刷写**的固件，它的核心使命只有一个——**在线刷写应用程序**。

> **一句话：Bootloader = ECU 的「恢复模式」，用于升级 Bootloader 之外的任何软件**

### 为什么需要 Bootloader？

ECU 出厂后，软件不可能一成不变：修复 bug、增加功能、适配新车型……但 ECU 刷写过程中如果刷到一半断电或通信中断，ECU 就会变「砖」。

**Bootloader 解决了这个问题：**
1. Bootloader 本身不参与应用功能，只负责刷写
2. Bootloader 放在独立的、受保护的 Flash 区域，**不可擦除**
3. 即使应用刷坏了，Bootloader 还在，可以重新刷

### ECU 的 Flash 布局

```
┌──────────────────────────────────────┐
│             Bootloader                │ ← 受保护，只读，不可擦除
│  (上电启动→检查是否需要刷写→跳转)       │
├──────────────────────────────────────┤
│          Bootloader 配置区            │ ← 刷写标志、版本信息
├──────────────────────────────────────┤
│          应用程序 (App)               │ ← 正常的 ECU 功能代码
│   (UDS 诊断、控制逻辑、通信栈…)         │
├──────────────────────────────────────┤
│       应用程序配置区 (NVRAM)           │ ← 标定参数、DTC 记录
└──────────────────────────────────────┘
```

### 上电流程

```
ECU 上电复位
    │
    ▼
Bootloader 启动
    │
    ├─ 检查刷写请求标志
    │   ├─ 有请求 ──► 进入刷写模式 (0x10 02 编程会话)
    │   └─ 无请求 ──► 检查 App 有效性
    │
    ├─ 检查 App 有效性
    │   ├─ 有效 ──► 跳转到 App 入口地址
    │   └─ 无效 ──► 等待诊断仪连接（防止变砖）
    │
    ▼
跳转到 App，正常运行
```

### 刷写流程（UDS + Bootloader 协同）

诊断仪通过 UDS 服务控制 Bootloader 执行刷写：

```
诊断仪                              ECU
  │                                  │
  ├── 0x10 02 (切编程会话) ────────► │  Bootloader 识别到编程会话
  ├── 0x27 (安全解锁) ────────────► │  验证密钥
  ├── 0x31 (擦除 Flash) ──────────► │  Bootloader 执行擦除
  ├── 0x34 (请求下载) ────────────► │  协商地址、大小
  ├── 0x36 (传输数据块) ──────────► │  Bootloader 逐块写入 Flash
  │     ... 重复多帧 ...              │
  ├── 0x37 (传输结束) ────────────► │  完成写入
  ├── 0x31 (验证 CRC) ────────────► │  校验完整性
  ├── 0x11 01 (复位 ECU) ─────────► │  ECU 重启，Bootloader 检查 App
  │                                  │  有效 → 跳转新 App
  │                                  │
```

**关键点：** Bootloader + UDS 就组成了一个完整的「刷写链路」。Bootloader 负责底层的 Flash 擦写，UDS 负责上层的命令协议。

### 常见 Bootloader 方案

| 方案                    | 说明                             | 适用              |
|:--------------------|:-----------------------------|:--------------|
| **OEM 自研 Bootloader** | 主机厂自己写，配合产线工具                  | 各大主机厂           |
| **Vector 方案**         | CANoe 刷写 + vFlash              | 开发测试阶段          |
| **EB tresos / KPIT**  | AUTOSAR 标准 Bootloader          | 符合 ASPICE 的车规项目 |
| **开源 UDS Bootloader** | 如 OpenBLT、STM32 UDS Bootloader | 学习/原型验证         |
| **MCU 厂商官方方案**        | NXP S32K Bootloader、TI TMS570  | 基于特定芯片          |

### 安全相关

- **安全解锁（0x27）**：防止非法刷写，密钥通常基于种子+算法，不同 ECU 密钥不同
- **签名验证**：刷写前 Bootloader 会验证固件的数字签名，防止刷入篡改固件
- **CRC 校验**：刷写完成后校验完整性
- **回滚保护**：升级失败时能回退到上一版本
- **安全启动 (Secure Boot)**：Bootloader 启动时逐级验证（Bootloader → App），防止篡改

### Bootloader 与 CICD 的关系

回到奇瑞 CICD 岗位，理解 Bootloader 对以下工作很关键：

1. **编译产物管理**：CI/CD 产出的 `.hex` / `.s19` / `.bin` 需要按 Bootloader 要求的分区地址来生成
2. **刷写脚本编写**：产线或 OTA 的刷写流程本质就是 UDS + Bootloader 对话的脚本化
3. **版本兼容性**：Bootloader 和 App 的版本需要匹配（Bootloader 升级通常需要特殊工具）
4. **失败重试机制**：CI/CD 中刷写失败的处理逻辑依赖 Bootloader 的安全机制（是否变砖、能否重刷）
